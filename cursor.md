# Проект: Mapvideo

## Описание
Веб-сервис для редакции телеканала. Пользователи создают анимированные карты
для новостных сюжетов в прямом эфире и скачивают готовое MP4/MXF видео.

## Домен и окружение
- Продакшн: https://mapvideo.gyhyry.ru
- ОС сервера: Ubuntu 24 LTS
- ВАЖНО: на сервере уже работает независимый проект maps.gyhyry.ru —
  не трогать его nginx конфиги и сервисы ни при каких обстоятельствах



## Принципы разработки

### 1. Думай перед тем как кодировать
- Не делай молчаливых предположений — если что-то непонятно, спроси
- Если существует несколько интерпретаций задачи — озвучь их, не выбирай молча
- Если есть более простой подход — скажи об этом
- Если запутался — остановись, назови что неясно и попроси уточнение

### 2. Простота прежде всего
- Минимум кода который решает задачу — ничего лишнего
- Никаких абстракций для одноразового кода
- Никакой "гибкости" или "расширяемости" которую не просили
- Никакой обработки ошибок для невозможных сценариев
- Если 200 строк можно заменить 50 — переписать
- Тест: сказал бы опытный разработчик что это overcomplicated? Если да — упростить

### 3. Хирургические изменения
- Трогай ТОЛЬКО то что нужно для задачи
- Не "улучшай" соседний код, комментарии или форматирование
- Не рефакторь то что не сломано
- Соблюдай существующий стиль кода даже если сам сделал бы иначе
- Если заметил мёртвый код — упомяни, но не удаляй без запроса
- КРИТИЧНО: не трогать конфиги nginx maps.gyhyry.ru — независимый проект
- Тест: каждая изменённая строка должна прямо вытекать из запроса пользователя

### 4. Выполнение через цели
- Определи критерии успеха до начала работы
- Трансформируй задачи в проверяемые цели:
  - "Добавь валидацию" → "Напиши тесты для невалидных данных, потом сделай их проходящими"
  - "Исправь баг" → "Напиши тест который воспроизводит баг, потом сделай его проходящим"
- Для многошаговых задач озвучивай план: Шаг → проверка: что проверяем
- LLM исключительно хорош когда крутится в цикле до достижения конкретной цели

## Формат разработки
- Разработка ведётся строго в соответствии со SPEC.md
- Работаем последовательно: task1.md → task2.md → ... → task9.md
- Следующий task файл открывается ТОЛЬКО после полного завершения текущего
- После завершения каждого task файла — обновить статус фазы в SPEC.md
- При возобновлении сессии (после обновления лимита токенов):
  1. Прочитать SPEC.md — найти текущую фазу (статус "В работе")
  2. Открыть соответствующий task*.md
  3. Найти последнюю заметку "## Заметка для следующей сессии"
  4. Продолжить с того места где остановились

## Технологический стек

### Бэкенд
- Runtime: Node.js 20+ с TypeScript
- Фреймворк: Express.js
- Рендер видео: Puppeteer + FFmpeg
- Тесты: Jest

### Фронтенд
- Фреймворк: React 18 + TypeScript + Vite
- Карта (подложка): MapLibre GL JS
- Анимации: PixiJS (WebGL рендеринг) + GSAP (управление таймлайнами)
- Запись превью: MediaRecorder API
- Тесты: Vitest + React Testing Library

### Инфраструктура
- Контейнеризация: Docker + Docker Compose
- Прокси: Nginx (уже установлен на сервере)
- SSL: Let's Encrypt (Certbot, уже настроен)
- Тайл-сервер: Martin (векторные тайлы из PostGIS)
- База данных: PostgreSQL 16 + PostGIS
- Геокодер: Photon (поиск и автодополнение адресов)
- Маршрутизация: OSRM (маршруты по реальным дорогам)
- Данные карт: OpenStreetMap (весь мир, planet-latest.osm.pbf)
- Импорт OSM данных: osm2pgsql

## Docker сервисы и порты (внутренние)
| Сервис     | Порт  | Описание                          |
|------------|-------|-----------------------------------|
| backend    | 3001  | Node.js API                       |
| frontend   | 3000  | React (dev) / nginx (prod)        |
| martin     | 3002  | Векторные тайлы                   |
| postgres   | 5432  | PostgreSQL + PostGIS              |

```bash
docker compose up -d                    # Запустить все сервисы
docker compose up -d backend frontend   # Только приложение
docker compose logs -f backend          # Логи бэкенда
```

```bash

docker compose up -d
```

- Импорты: ES modules (import/export), не CommonJS (require)

## Архитектурные решения

### Анимационный движок
- PixiJS рендерит все анимации поверх карты через WebGL
- GSAP управляет таймлайнами и синхронизирует анимации с длительностью видео
- При серверном рендере GSAP таймлайн прокручивается покадрово
- Подробности: task5.md, task8.md

### Рендер видео
- Puppeteer запускается с флагами --use-gl=angle --use-angle=swiftshader
  (обязательно для WebGL в headless окружении)
- Подробности: task8.md

### Карта без логотипов
- Собственный OSM сервер — никаких сторонних логотипов
- MapLibre GL JS — open source, BSD лицензия

### Данные карты
- Используется полный мировой датасет: planet-latest.osm.pbf (~80GB)
- Файл скачивается один раз и хранится в osm-data/ на сервере
- Импорт занимает несколько часов, выполняется один раз

## Важные ограничения
- osm-data/ НЕ коммитить в git
- .env файлы НЕ коммитить в git
- Не трогать nginx конфиги maps.gyhyry.ru
- При добавлении Docker сервисов проверять что порты не конфликтуют

## Правило обновления прогресса
- При начале фазы — обновить статус в таблице SPEC.md на "🔄 В работе" и обновить поле "Текущая фаза"
- После завершения фазы — обновить статус на "✅ Сделано"
- При начале задачи внутри фазы — отметить [~] в соответствующем task*.md
- По ходу процесса разработки обновлять "## Заметка для следующей сессии" в соответствующем task*.md
- После завершения задачи — отметить [x] немедленно, не накапливать
- Перед завершением сессии (при любой остановке) — обязательно записать заметку
  в конец текущего task*.md: "## Заметка для следующей сессии: ..."
  и обновить поле "Последняя остановка" в таблице SPEC.md
