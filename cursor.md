# Проект: Mapvideo

## Описание
Веб-сервис для редакции телеканала. Пользователи создают анимированные карты
для новостных сюжетов в прямом эфире и скачивают готовое MP4/MXF видео.

## Домен и окружение
- Единственное окружение: https://mapvideo.gyhyry.ru (Ubuntu 24 LTS) — это тестовый проект, отдельной dev-инсталляции не делаем
- Разработка ведётся прямо на сервере через Cursor Remote-SSH
- ВАЖНО: на сервере уже работает независимый проект maps.gyhyry.ru — не трогать его nginx конфиги и сервисы ни при каких обстоятельствах

## Окружения
- Одно окружение: **сервер**. Репозиторий — в `~/mapvideo` (внутри домашней папки пользователя на сервере)
- OSM по умолчанию: **Россия** (`russia-260419.osm.pbf`, ~3-4 ГБ, импорт 20-40 мин). Файл уже лежит в `osm-data/` на сервере
- Переход на весь мир — одна переменная `OSM_PBF_FILE` в `.env`, никаких изменений кода
- Dev-режим запускается через `docker compose -f docker-compose.yml -f docker-compose.dev.yml up -d` — пробрасывает внутренние порты на localhost сервера, чтобы можно было проверять сервисы по отдельности через SSH-туннель VS Code (он автоматически форвардит порты)

## Принципы разработки

### 1. Думай перед тем как кодировать
- Не делай молчаливых предположений — если что-то непонятно, спроси
- Если существует несколько интерпретаций задачи — озвучь их, не выбирай молча
- Если есть более простой подход — скажи об этом
- Если запутался — остановись, назови что неясно и попроси уточнение

### 2. Простота прежде всего
- Минимум кода который решает задачу — ничего лишнего
- Никаких абстракций для одноразового кода
- Никакой "гибкости" или "расширяемости" которую не просили
- Никакой обработки ошибок для невозможных сценариев
- Если 200 строк можно заменить 50 — переписать
- Тест: сказал бы опытный разработчик что это overcomplicated? Если да — упростить

### 3. Хирургические изменения
- Трогай ТОЛЬКО то что нужно для задачи
- Не "улучшай" соседний код, комментарии или форматирование
- Не рефакторь то что не сломано
- Соблюдай существующий стиль кода даже если сам сделал бы иначе
- Если заметил мёртвый код — упомяни, но не удаляй без запроса
- КРИТИЧНО: не трогать конфиги nginx и сервисы maps.gyhyry.ru — это независимый проект
- Тест: каждая изменённая строка должна прямо вытекать из запроса пользователя

### 4. Выполнение через цели
- Определи критерии успеха до начала работы
- Трансформируй задачи в проверяемые цели:
  - "Добавь валидацию" → "Напиши тесты для невалидных данных, потом сделай их проходящими"
  - "Исправь баг" → "Напиши тест который воспроизводит баг, потом сделай его проходящим"
- Для многошаговых задач озвучивай план: Шаг → проверка: что проверяем
- LLM исключительно хорош когда крутится в цикле до достижения конкретной цели

## Формат разработки
- Разработка ведётся строго в соответствии со SPEC.md
- Работаем последовательно: task1.md → task2.md → ... → task9.md
- Следующий task файл открывается ТОЛЬКО после полного завершения текущего
- После завершения каждого task файла — обновить статус фазы в SPEC.md
- При возобновлении сессии (после обновления лимита токенов):
  1. Прочитать SPEC.md — найти текущую фазу (статус "В работе")
  2. Открыть соответствующий task*.md
  3. Найти последнюю заметку "## Заметка для следующей сессии"
  4. Продолжить с того места где остановились

## Технологический стек

### Бэкенд
- Runtime: Node.js 20+ с TypeScript
- Фреймворк: Express.js
- Рендер видео: Puppeteer + FFmpeg
- Тесты: Jest

### Фронтенд
- Фреймворк: React 18 + TypeScript + Vite
- Карта (подложка): MapLibre GL JS (^4.7)
- Анимации: PixiJS **7.4.2** (не v8 — v8 имеет async init, ломает примеры в task5/task6) + GSAP (без премиум плагинов)
- HTTP: axios с `withCredentials: true` (для httpOnly session cookie) + интерцептор для `X-CSRF-Token`
- Тесты: Vitest + React Testing Library

### Рендер видео — архитектура
- **Только серверный** рендер через Puppeteer + FFmpeg. Браузерный MediaRecorder не используется (нужно работать при закрытой вкладке)
- Кадры из Puppeteer передаются в FFmpeg stdin через pipe (image2pipe) — не записываются на диск
- Состояние рендера хранится в PostgreSQL (`render_jobs`), переживает рестарт бэкенда
- Движение иконок транспорта по пути — собственная функция интерполяции (GSAP MotionPath премиум, не используем)

### Инфраструктура
- Контейнеризация: Docker + Docker Compose
- Прокси: Nginx (уже установлен на сервере)
- SSL: Let's Encrypt (Certbot, уже настроен)
- Тайл-сервер: Martin (векторные тайлы из PostGIS)
- База данных: PostgreSQL 16 + PostGIS
- Геокодер: Photon **через публичный сервис photon.komoot.io** (прокси на бэкенде). Если упрёмся в rate limits или сервис ляжет — поднимем self-hosted worldwide индекс (~75 ГБ) как docker-сервис. В docker-compose Photon **не поднимаем**
- Маршрутизация: OSRM (маршруты по реальным дорогам)
- Данные карт: OpenStreetMap (по умолчанию Россия: russia-260419.osm.pbf, при необходимости можно переключить на planet-latest.osm.pbf)
- Импорт OSM данных: osm2pgsql

**КРИТИЧНО:** для разработки и прода используется только **хостовой** `nginx`. Запуск `nginx` как Docker-сервиса запрещён.

## Docker сервисы и порты (внутренние)
| Сервис     | Порт (внутри compose) | Bind на хосте     | Описание                          |
|------------|-----------------------|-------------------|-----------------------------------|
| backend    | 3001                  | 127.0.0.1:3001    | Node.js API                       |
| frontend   | 3000                  | 127.0.0.1:3000    | React (dev) / nginx статика (prod)|
| martin     | 3000                  | 127.0.0.1:3002    | Векторные тайлы                   |
| postgres   | 5432                  | 127.0.0.1:5432    | PostgreSQL + PostGIS              |
| osrm       | 5000                  | 127.0.0.1:5000    | Маршрутизация                     |

**КРИТИЧНО: Nginx в docker compose НЕТ.** Снаружи 80/443 слушает **только хостовой nginx сервера** (тот же, что обслуживает maps.gyhyry.ru). Он проксирует `mapvideo.gyhyry.ru` на сервисы по `127.0.0.1:PORT`. Photon в compose тоже нет — геокодер работает через публичный https://photon.komoot.io/api, вызов идёт с бэкенда.
`nginx/mapvideo.gyhyry.ru.conf` в репозитории — шаблон/источник правды для установки в `/etc/nginx/sites-available/` на хосте.

## Архитектурные решения

### Анимационный движок
- PixiJS рендерит все анимации поверх карты через WebGL
- GSAP управляет таймлайнами и синхронизирует анимации с длительностью видео
- При серверном рендере GSAP таймлайн прокручивается покадрово
- Подробности: task5.md, task8.md

### Рендер видео
- Puppeteer запускается с флагами --use-gl=angle --use-angle=swiftshader
  (обязательно для WebGL в headless окружении)
- Подробности: task8.md

### Карта без логотипов
- Собственный OSM сервер — никаких сторонних логотипов
- MapLibre GL JS — open source, BSD лицензия

### Данные карты
- По умолчанию используется **российский экстракт**: `russia-260419.osm.pbf` (~3-4 ГБ)
- Путь и имя файла задаются переменной `OSM_PBF_FILE` в `.env` — для перехода на весь мир достаточно скачать `planet-latest.osm.pbf` и обновить переменную
- Файл хранится в `osm-data/` (не в git)
- Импорт России: 20-40 минут. Импорт всего мира: несколько часов
- Источник fallback-загрузки (если локального файла нет): https://download.geofabrik.de/russia-latest.osm.pbf

## Важные ограничения
- osm-data/ НЕ коммитить в git
- .env файлы НЕ коммитить в git
- Не трогать nginx конфиги maps.gyhyry.ru
- При добавлении Docker сервисов проверять что порты не конфликтуют

## Переменные окружения (.env)
```
# PostgreSQL
POSTGRES_USER=mapvideo
POSTGRES_PASSWORD=<сгенерировать случайный>
POSTGRES_DB=mapvideo
POSTGRES_GIS_DB=gis

# Админ
ADMIN_PASSWORD=<сгенерировать случайный>
# ADMIN_PASSWORD_ROTATE_ON_BOOT=false   # true только для принудительной ротации на старте

# Cookies (сессионный токен — opaque random, отдельный секрет для подписи не нужен)
COOKIE_SECURE=true

# OSM данные
OSM_PBF_FILE=russia-260419.osm.pbf       # по умолчанию — Россия
# OSM_PBF_FILE=planet-latest.osm.pbf     # переключить если нужен весь мир

# Геокодер — публичный Photon (если упрёмся в rate limit, поднимем self-hosted)
PHOTON_URL=https://photon.komoot.io/api

# Пути внутри контейнеров
OSRM_DATA_PATH=/data/region.osrm    # использовать в osrm command в compose вместо hardcode

# Окружение
NODE_ENV=production   # одно окружение — сервер; для dev-режима docker-compose.dev.yml + NODE_ENV=development
```

Bind `127.0.0.1:5432:5432` для PostgreSQL относится к dev/test схеме.
На production внешняя публикация PostgreSQL не нужна.
